The tweak in this repo is somewhat sophisticated. Advice on simplification is welcome.

The motivation is to combine the blazing-fast tex compilation on WSL with seamless forward and backward search experience.

Versions

The solution is valid undering these settings (64 bit version by default)

  • Windows 11 22H2
  • WSL Ubuntu 22.04.2 LTS
  • VSCode version 1.78.2
  • SumatraPDF v3.4.6
  • TeX Live 2023
  • LaTex Workshop Plugin v9.11.4

Rough Intro to Setting-Up

I am going to skip installing WSL, VSCode and SumatraPDF on Windows, as this step is relatively simple. Things to notice:

  1. SumatraPDF is simply installed on Windows, no special care needed
  2. Make sure to install the WSL extension , and also the LaTex Workshop extension in WSL

Installing Tex Live 2023 in WSL

It should be stressed that Tex Live is installed on WSL. The steps here refer to the section of WSL in this document

  1. Download full Tex Live ISO from mirror or torrent
  2. Double click on the ISO in File Explorer to be mounted on the Windows System. Check the name of the mounted image (eg. U:\, F:\), suppose that it is U:\
  3. Inside WSL terminal
  4. Cleaning up
  5. Handle PATH variable. Append this fragment to your favorate profile (eg. .profile, .bashrc) using your favorate editor
  6. Handle fonts

WSL VSCode settings.json

Note:

  1. This is different from the settings.json for Windows VSCode. The file is located at /home/<ur_usrname>/configs/vscode/settings.json
  2. Change <ur_usrname> to your WSL username
  3. /home/<ur_usrname>/.local/bin can be replaced with arbitrary path, it is just my convention here. Stick to using abs path.
  4. forward.sh and backward.sh are the two utility scripts that are used to forward and backward search. They are explained in the next section.
  5. The rpsync.sh is a utility script that is used to modify synctex file
  6. Attend to the comments in the json file should you have questions
  7. After you have setup settings.json, you should be able to compile and run simple LaTex documents. At this stage, resort to the simple recipe that only uses one call to xelatex. Switch to using xelatex recipe after completing the following section on forward.sh and rpsync.sh

Setting-up Symbolic Link on Windows Filesystem

Most of the complexity of using synctex with WSL has to do with path issues. The most salient is with SumatraPDF's default behavior of handling WSL paths (adding C:\) as prefix.

To circumvent this, I did two things.

  1. Create a symbolic link on Windows filesystem that points to the dir that I am working on in WSL.
  2. Create a script that automatically modified synctex generated files at compilation time.
  • Reason 1: SumatraPDF would access the pdf using Windows path, therefore precluding path issues. The major factor for performance is whether Tex Live is installed and compiled on WSL or Windows. Therefore, this tweak is acceptable
  • Reason 2: The LaTex Workshop plugin makes it handy to chain and automate the whole process

Steps

  1. Go to the working directory on WSL
  2. Get the windows path for this folder
  3. Open Commandline Terminal in admin mode (instead of Powershell)
    In my case, pwd on WSL gives /home/<ur_usrname>/tex_notes. I linked "C:\Users\<ur_usrname>\Documents\tex_notes" to it.

rpsync.sh and forward.sh

One thing to mention is that these scripts assume that the symbolic links on Windows are created under one common dir, with different links pointing to different project dirs on WSL. ### rpsync.sh It is obvious that this script automatically patches the paths in synctex.gz

forward.sh

It follows that this script automatically manipulates the paths passed in from LaTex Workshop plugin

Putting them in place

In my case, I put the scripts under $HOME/.local/bin. You could choose any location to your heart's content. Remember to modify setting.json accordingly

Conclusion

The main idea is to leave the bulk of the compilation in WSL, and automate the rest of the process through lightweight scripts and VSCode extension to simulate the env under Windows.
